# Option value of reclaimed stock — an ML design note (C3, designed, not built)

This is a **design document only**. No model described here is wired into the matching pipeline, and
none is proposed to be. It sets out how the dormant `ml/` layer could earn a *non-circular* job —
estimating each stock item's **option value** (its probability of a better future match) — and how
that estimate would *calibrate*, never *replace*, the experimental scarcity weight from phase C2
(`--reserve`, `match.optimize._apply_reserve`). It is the third leg of the stock-stewardship plan in
[OVERVIEW.md §13](OVERVIEW.md): **A/B** are accounting and single-project knobs,
**C1** (portfolio matching) is the principled multi-project optimiser, **C2** is a cheap
single-project proxy for the option value of holding stock back, and **C3** — here — is how you would
put a defensible *number* on that option value instead of a hand-tuned weight.

## Why this is the honest ML job (and the surrogate was not)

The repo is deliberate about where machine learning may and may not sit (DESIGN_PRINCIPLES.md rule 3): the
certified result path is entirely deterministic — EN 1993-1-1 checks → carbon accounting → MILP — and
nothing learned is ever allowed to *gate* a structural decision. The existing capacity surrogate
([`experiments/ml/surrogate.py`](../experiments/ml/surrogate.py)) makes that point by counter-example: its
test R² ≈ 1.0 is **circular**, because its labels are produced by the EN 1993 checker it imitates, so
the score only shows it can reproduce a function we already compute exactly. It predicts nothing new.

Option value is different. It is a genuinely **unknown** quantity — it depends on *future demand the
tool has never seen* — so there is no deterministic oracle to be circular against, and a model that
estimates it is adding information rather than echoing it. Crucially, the output is **decision
support, never a gate**: at most it tunes a soft, opt-in score penalty (C2) or annotates the
disposition advisory (A2). A member is never refused reuse because a model thought it might be useful
later; the deterministic checks and the booked-carbon objective remain the only authorities.

## What "option value" means here

For a stock item *d* (a reclaimed member of a given section family, length and condition), define its
option value as the **expected net CO₂ it would save in its best future use, relative to spending it
on a slot it is over-qualified for today**:

```
OV(d) = E_future[ best CO2 saving d achieves across plausible future demand ]
        - ( CO2 saving d achieves on the current-project slot it would be spent on )
```

A high `OV(d)` is the formal version of "don't put the solid column in a partition-wall slot —
something will need it." It is exactly the quantity C2 approximates with a crude, demand-free proxy:
`scarcity_f = (# current slots family f can serve) / (# donors of family f)`, penalised only on slots
another family could also fill. C2 is honest about being a proxy (see its docstring); C3 replaces the
proxy's hand-set weight with an estimate grounded in a **demand distribution**.

## The model

### Inputs (features per stock item)

Drawn entirely from data the tool already has on a `SupplyItem` / audited donor:

- section family and its section properties (`A`, `Wpl,y`, `Iy`, mass/m, depth, flange thickness);
- length, and length *class* (how many standard spans it can be cut to);
- material grade and the audit knockdown (a derated heavy member is worth less held);
- a standardization proxy — how common the family is in the wider reclaimed-steel market, not just
  this stock (the [`experiments/ml/reuse_score.py`](../experiments/ml/reuse_score.py) standardization term is
  the in-stock version of this).

### Label (the target to learn)

`OV(d)`, generated by **simulation over a demand distribution**, not by a closed form:

1. Fit or assume a distribution over future projects — span ranges, load levels, grade mix, project
   size, and how often each demand *section class* appears. The bootstrap source is the existing
   synthetic sweep ([`synthetic.py`](../experiments/ml/synthetic.py) already enumerates
   section × grade × span × load through the real EN 1993 checks), refined with the case-study demand
   models (`data/case_study/demand.json` and the bundled sample) as empirical anchors for what real structures
   actually ask for.
2. Monte-Carlo *K* synthetic future projects from that distribution.
3. For each draw, run the **existing deterministic matcher** (`match.optimize.match`) with *d* present
   and again with *d* absent; the difference in booked net CO₂ is *d*'s realised value in that future.
4. `OV(d)` is the expectation (mean, or a chosen quantile for risk aversion) over the *K* draws.

This makes the deterministic EN 1993 + carbon + MILP stack the **label oracle** — the model learns to
*predict the optimiser's own verdict over a demand distribution* quickly, so a planner can rank a
yard of hundreds of donors without re-running *K* full matches each time. That is a real
speed-and-generalisation job, and it never overrides the deterministic check: when a specific project arrives,
the actual MILP is what runs.

### Estimator

A gradient-boosted regressor (the `xgboost` dependency is already an optional extra and
`experiments/ml/train.py`/`surrogate.py` show the pattern) or any interpretable model — feature importances and
partial-dependence plots matter more than squared error here, because the output must be *explainable*
to an engineer deciding to store steel. Quantile regression gives a conservative (e.g. 25th-percentile)
option value for risk-averse storage decisions.

## How it would calibrate C2 — and only calibrate it

C2's penalty today is `reserve_w × scarcity_norm_f × mass_used × saved_per_kg`, with `reserve_w` and
the `scarcity_f` proxy both hand-chosen. The model feeds two things back, **without changing the
mechanism**:

1. **Replace the proxy `scarcity_f` with a learned per-item `OV(d)`** (normalised the same way), so the
   penalty reflects estimated future value rather than a current-project feasibility ratio. The shape
   of the term — score-only, applied only on slots another family can serve, never touching booked
   CO₂, default off — is unchanged. `verify_match` would re-derive it from weights exactly as now.
2. **Set `reserve_w` by calibration instead of by hand**: choose the weight at which the soft penalty
   reproduces, on held-out synthetic portfolios, the allocation the *full C1 portfolio optimiser*
   would have made with the future projects visible. C1 is the ground truth (it actually sees the
   other projects); C2+OV is the single-project approximation; the model picks the weight that makes
   the approximation track the truth. That gives `reserve_w` a defensible value and an error bar,
   rather than a guess.

The principled tool remains C1: when you *have* the other demand models, optimise over them directly.
C3 is for the realistic case where future demand is a distribution, not a set of files.

## What a real deployment would need

The synthetic bootstrap is enough to *build and test* the method; trusting its numbers in practice
needs data the project does not yet have:

- **A real demand distribution** for the deploying organisation — a portfolio or pipeline of past and
  forecast projects (spans, loads, grades, section mix), not a generic sweep. Option value is
  organisation- and market-specific; a steel stockholder and a single developer have different futures.
- **Realised reuse outcomes** to validate the simulated labels — did stock that scored high option
  value actually get reused later, and at what saving? Until then the labels are model-on-model.
- **Holding-cost parameters** so the comparison is honest: storage is *not* free (yard rent, double
  handling, transport, the risk the section is never needed, degradation in store). The plan already
  insists these be explicit parameters; option value must be netted against them, or the model will
  always advise hoarding.
- **Market standardization data** — how liquid each section family is in the reclaimed market — to
  feature-engineer the standardization input beyond the current single stock.

## Scope guardrails (restating the decision)

- **Out of the pipeline.** No model output gates, filters, or ranks a *certified* match. The
  deterministic EN 1993 check and the booked-carbon MILP objective stay the sole authorities
  (DESIGN_PRINCIPLES.md rule 3).
- **Decision support only.** Permitted touch-points are: tuning the opt-in C2 `reserve_w`/scarcity
  term (which is itself score-only and default off), and enriching the A2 stock-disposition advisory
  (store / re-roll / recycle) with an estimated option value column. Both are advisory.
- **Honest about uncertainty.** Wide error bars, conservative quantiles, and holding costs netted in —
  an option-value estimate that ignores storage cost or its own variance is worse than the honest
  hand-tuned proxy it would replace.

This note exists so the idea is recorded with its guardrails; it is **designed, not built**.
